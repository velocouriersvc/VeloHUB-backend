# Round 19 Fixes - 2026-07-21

Tester feedback (Maxine, Bernard, Solomon). Priority was "make the app work first", so this
round ships as **PR 1 (P0 + P1)** to unblock testers. The fee/tax/vehicle-tier specs are PR 2.

## P0 - Core flows

### P0-1 Store-pickup handshake + correct code distribution
- `merchant-service.verifyPickupCode`: the code compare is now case-insensitive and
  whitespace-tolerant (matches `pickup-code-service.verify`). A correct code no longer fails on
  casing.
- App `order-details.tsx`: scanned/typed codes are normalized (uppercase + trim) before submit.
- `order-service`: DELIVERY orders no longer mint a customer pickup code (dead data that made
  testers show the wrong code). The merchant-side handshake for deliveries uses the driver's
  4-digit PIN; the customer only ever receives the **delivery** code, which they hand to the driver
  at drop-off. `sendOrderCodesNotification` now sends exactly one code per order type with clear copy.

### P0-2 Completion code reaches the customer
- `service-booking-service.updateBookingStatus`: when the provider ACCEPTS / SCHEDULES / starts a
  booking, the customer is notified with the completion code ("Give your provider this code when the
  job is done").
- App `services/index.tsx`: the completion code is rendered on the customer's booking card (tap to
  copy) once accepted.
- `completeBooking` code compare is now case-insensitive / trimmed.

### P0-3 Product options multi-select
- App `marketplace/product/[id].tsx`: options are now multi-select where `maxSelections != 1`
  (e.g. Waakye extras: Egg + Goat meat + Wele together), with a cap when `maxSelections > 1`, price
  summed across all picks, and a checkbox/radio indicator. Backend already accepted arrays.
- `cart-service.validateRequiredCustomizations` now also enforces the `maxSelections` cap.

### P0-4 Ride cancellation refunds to source
- `ride-service.cancelRide`: any cancellation of a PAID (prepaid card/momo) ride now refunds the
  fare via `paymentService.refundToSource` (Paystack to the original method, wallet fallback) and
  notifies the customer. Cash rides collected nothing, so nothing is refunded.

### P0-5 Same-day bookings greyed out
- `booking-calendar.tsx`: earliest bookable day is now tomorrow; today and past days are disabled.

### P0-6 Copy address + provider "Book a Velo ride"
- Added `expo-clipboard` (native dep - requires a fresh build).
- Customer booking card: tap the service/provider address to copy it (paste as a ride destination).
- Provider `service-bookings.tsx` (out-call): "Copy address" and "Book a Velo ride here" (prefills
  the ride destination with the customer's location).

## P1 - Operational

### P1-1 Finances "See All" transactions error
- `merchant-service.getTransactions` now returns the `{ data, meta: { total, page, limit,
  totalPages } }` shape the app's `PaginatedTransactionsResponse` already declares (was
  `{ transactions, ... }`, which crashed the activity screen). App reader guarded defensively.

### P1-2 Real Paystack payouts
- `paystack-provider`: added `createTransferRecipient` (POST /transferrecipient) and
  `initiateTransfer` (POST /transfer); declared as optional methods on the provider interface.
- `wallet.ts`: new nullable `paystackRecipientCode` column (auto-added via synchronize).
- `wallet-service`: `ensurePayoutRecipient` (creates/caches a recipient from momo/bank details,
  Ghana momo providers mapped to Paystack bank codes) and `initiatePayoutTransfer`. Also a shared
  `requestPayout` used by drivers.
- `merchant-service.requestPayout`: creates/reuses the recipient at request time (non-fatal).
- `admin-service.approvePayout`: now performs the real transfer to the stored recipient and only
  marks the payout completed on success; a provider failure keeps it pending with the error recorded.
- Driver payout: new `POST /wallet/request-payout` (buyer/driver roles). The driver app's payout
  screen (previously a pure mock) now collects the momo provider + number and calls the endpoint.

### P1-3 Live balance reset (executed)
- Zeroed all driver + merchant wallet balances on prod (customers untouched, transaction history
  preserved). 4 distinct wallets cleared (drivers 2327.32 + merchants 3266.23 across overlapping
  roles); verified 0 non-zero driver/merchant wallets remain.
