# Round 21 - 2026-07-23

Tester feedback after the Round 20 deploy: services chat "still not working", the Android delivery
code crash, and three package-request requirements. Root-caused against the code and, where possible,
verified against the live production server during planning.

## Slice 0 - Stop debugging blind

`AppErrorBoundary` used to hide the real error in production and only `console.error` it (no Sentry),
so the Android delivery crash was invisible. It now renders the real `error.message` plus a "Copy
error details" button (message + component stack). Any future crash is diagnosable from a tester's
screenshot.

## Slice 1 - Deliveries: driver code handshake that works on Android

The screen recording showed the driver on the live-map screen (`navigation.tsx`) reaching drop-off,
the "DROP-OFF PIN" showing "-", and "Confirm Delivery" crashing into the error boundary. Two root
problems:

- **Functional:** for a delivery, `navigation.tsx` "Confirm Delivery" was a no-op that only advanced
  the local UI. It never collected/verified the customer's code and never called the backend, so the
  delivery was never completed, settled or paid out. There was also no driver-pickup-code display.
- Now the screen displays the driver's pickup code at pickup (for the merchant to key in), and at
  drop-off has a real code `TextInput` that calls `verifyDeliveryCode` + `completeDelivery`
  (settlement + payout). The critical path uses plain Views (no reanimated `entering` animation, the
  prime New-Arch/Android crash suspect) and hardened `Number(x ?? 0).toFixed()` formatting.

Backend:
- `verifyDeliveryCode` now accepts a `PICKED_UP` order. After the merchant releases a package the
  order sits at PICKED_UP, so the driver could previously never enter the customer's code.
- The proof-of-delivery photo is no longer mandatory to complete: the customer's code is the handover
  proof (the flow is code -> delivered -> payout, no photo step). A missing photo must never block
  the driver's settlement.
- Post-verify side effects (audit, socket) are best effort, so a notification failure never reports a
  false "invalid code" after the handover committed.
- The active-delivery response includes the customer's name/phone for the driver's drop-off screen.

Also: `order-details` reject used `Alert.prompt`, which is iOS-only and silently did nothing on
Android; it now uses a cross-platform modal.

## Slice 2 - Chat opens once paid

Chat was verified fully working on the backend during planning (GET 200 / POST 201 on a live
contactable booking; the table exists; `synchronize` is on). "Still not working" was the Round 20
accept-gate: a tester on a not-yet-accepted booking saw no chat button and a blank chat.

- Chat + calling now open once the booking is PAID (status `requested`), not only after acceptance,
  so both sides can discuss up front. Only `awaiting_payment` and terminal states are closed. One line
  in `CONTACTABLE_STATUSES`.
- The chat screen now surfaces load errors (e.g. the gate reason) instead of rendering a dead empty
  chat, and hides the composer when the conversation cannot load.

## Slice 3 - Packages (delivery Rides): prepaid, real codes, no sharing

A package is a `Ride` with `type = DELIVERY`.

- **Prepaid only:** `requestRide` rejects a DELIVERY that is not card/momo, so a package is only
  broadcast to drivers after the charge confirms. The VeloSend payment step hides cash
  (`PaymentSelection` gained an opt-in `hideCash` prop; passenger rides keep cash).
- **Real codes:** the package reuses the coded-ride pickup flow (`requireCode` + `pickupCode` = the
  sender's code the driver enters to start) and adds a `packageDeliveryCode` the recipient gives the
  driver at drop-off. `completeRide` requires and verifies that code before settling. Both codes are
  sent to the sender at dispatch and shown on their tracking screen (pickup code + a delivery code to
  relay to the recipient). The driver payload strips both codes and exposes a `requirePackageDelivery`
  flag; the driver enters the drop-off code inline on the nav screen.
- **No ride-sharing:** `requestRide` never persists `sharedContacts` for a DELIVERY (VeloSend already
  has no share UI; this makes it a hard backend rule).
