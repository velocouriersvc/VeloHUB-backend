# Fixes - 2026-07-11 (round 4)

## Chat texts not crossing (both directions)  [ROOT CAUSE FIXED, app]
The driver socket hook derived its origin with `.replace('/api','')`, leaving `/v1` on the
URL, so the driver connected to the non-existent `/v1/drivers` namespace and was rejected.
The driver therefore never joined their personal room (customer->driver lost) and their
emits never reached the server (driver->customer lost) - while REST and push (used by
everything else) kept working, exactly matching the report. Fixed to strip `/api/v1` like
the customer hook. Also:
- Customers now REJOIN the ride/order room on socket reconnect (rooms are lost on every
  reconnect; previously a network blip silently killed driver->customer delivery forever).
- Drivers get an optimistic echo of their own sends (like customers).
- NEW: chat messages now trigger a push + in-app notification (`message_received` type,
  synced into the PG enum on boot) to the recipient; driver taps route to the driver rides
  screen (`screen: 'driver-rides'`).

## Error after completing a ride (money already settled)  [FIXED, backend]
`completeRide` ran unguarded Redis cleanup AFTER settlement; a Redis hiccup returned 400
("Update failed... Please try again") even though the wallet credit + payment alert had
already gone out. Post-settlement side effects are now best-effort (logged, never thrown).

## Driver dashboard stuck at 0 completed / 0.0 rating  [FIXED, backend + app]
`GET /driver/stats` read the DriverStats table, which is only written when a customer
submits a rating - completing rides never bumped it. The endpoint now computes live:
completed-ride count + SUM(driverPayout) + avg rating from the ratings table (customer
payloads never receive earnings). The dashboard also refreshes on focus (was mount-only).

## PIN verification optional for deliveries  [DONE end-to-end]
- `Order.requireDeliveryCode` (default false). Delivery PIN is generated ONLY when the
  customer flips the new "Require delivery PIN" toggle at checkout (rides already had
  their toggle).
- Driver "Confirm Delivery": coded orders keep the PIN prompt; non-coded orders complete
  via the existing code-free `completeDelivery` path with a confirm dialog.
- PIN in push: order code notifications already carried codes; ride arrival pushes now
  include the pickup code when the rider opted in ("Your pickup code: X").

## Live tracking notification (widget-like, Uber/Bolt style)  [DONE, app]
`utils/ride-tracking-notification.ts`: ONE local notification with a fixed identifier that
updates in place at each stage (searching -> driver assigned -> on the way -> arrived ->
trip in progress) with the driver's name + plate; sticky on Android; dismissed on
completion/cancellation. Driven from the existing rideStatus socket effect. No native
modules; ships with the EAS build.

## Velo Comfort -> Velo Priority, Velo XL -> Velo Premium  [DONE]
US/CA tier names aligned with the global names in `constants.ts`; backend fare-service
US/CA suv displayName now "Velo Premium".

## In-app notifications visible  [DONE, app]
Home header now has a bell icon (routes to the notifications screen) with a live unread
badge fed by the existing `unreadCount` from GET /notifications, refreshed on focus.

## Driver + vehicle info  [COMPLETED with real photo]
Name, plate, vehicle color/model, live rating + trip count were already end-to-end. Added
the missing piece: `DriverProfile.photoUrl` - the driver's avatar upload now persists to
the backend and customers see the real photo on the tracking card (initials fallback).
