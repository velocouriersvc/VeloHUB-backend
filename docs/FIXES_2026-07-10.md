# Fixes - 2026-07-10 (round 3)

Frame analysis of the 5 test videos first: the DRIVER videos run the NEW build (voice
toggle + chat button visible) but the CUSTOMER Android videos run the OLD build (old
"En Route / TO DESTINATION" headline and the old local-only chat where "hello" renders
only on the sender's phone). So "driver does not receive messages" and "code doesn't
generate at customer side" are largely the stale customer build - those fixes exist and
ship with the next EAS build. The rest were real bugs, all fixed below.

## 1. Receipt shown before the ride completes  [FIXED, app]
`mapStatusToStep` mapped backend `ongoing` -> the "Trip Completed / View Trip Receipt"
modal the moment the trip STARTED. New mapping: accepted/awaiting_payment/paid/
driver_enroute -> found (driver->pickup route, "To Pickup" ETA); ongoing -> en-route
(trip-in-progress view, "Heading to Location"); completed -> Trip Completed modal ->
receipt. Also the 10s simulated-arrival timer now only runs for the Velo Send flow;
plain rides advance purely on backend status.

## 2. "I have arrived" missing at the destination / stuck on "In Progress"  [FIXED, app]
The driver's Rides & Activity card locked into a static "In Progress" badge after
arrival, with no way to start or complete. The card now carries the full status-driven
ladder: I've Arrived -> Start Trip (coded rides route to the Navigation code modal) ->
Complete Trip, and refreshes on screen focus. The Navigation screen already had the full
ladder; its labels are now ride-aware ("Start Trip", "Arrived at Destination",
"Complete Trip") instead of delivery wording ("Start Delivery", "Confirm Delivery").

## 3. Dead "SCAN CODE -" box on the driver side  [FIXED, app]
The nav screen showed an ENTRY CODES box reading `pickupCode`, which the backend
deliberately never sends to drivers -> always a dash with no input. That box is now
deliveries-only. Coded rides instead show a hint ("This rider set a pickup code - ask
them for it") and the code is keyed into the existing modal on Start Trip. Reminder:
ride codes are OPT-IN at booking (the toggle on the ride summary); no toggle = no code
anywhere, by design.

## 4. Driver saw "Customer / Passenger" instead of the real name  [FIXED, backend]
`customerName` was flattened from `customer.fullName`, but the User entity has no name
column (names live on BuyerProfile/UserProfile). The driver active-ride query now joins
both profiles and resolves `buyerProfile.fullName ?? userProfile.fullName`.

## 5. Status / live tracking "doesn't show" on the customer side  [FIXED, app]
`app/orders/track.tsx` (the LIVE TRACKING screen reachable from history/menu) hardcoded
every ride timeline stamp to "--:--" and drew no driver marker. It now reads the real
timestamps (createdAt/acceptedAt/startedAt/completedAt), treats awaiting_payment/paid as
"accepted" for progress, and renders a live driver marker from the ride-room
`driver:location` stream.

## 6. Android: can't scroll to see driver info  [FIXED, app]
Root cause beyond the earlier handle-only-drag fix: the ride sheet is screen-height but
translated down, so ~42% of the inner ScrollView's frame sits BELOW the screen at the
tracking snap - the bottom content could never scroll into view. The tracking and
summary ScrollViews now pad by exactly that off-screen portion (from the sheet
constants) and set `nestedScrollEnabled` for Android.

## 7. Chat hardening  [IMPROVED, app]
Verified end-to-end: relay rooms line up (ride.driverId = User id = the driver's socket
room), so customer->driver delivery works on current builds. Hardened anyway:
- Customer sends now append optimistically (server echo replaces the local entry).
- The driver's Rides & Activity card gets a chat button + unread badge (previously chat
  existed only on the Navigation screen, which drivers may never open).
- `RideResponse` now types `requireCode`/`pickupCode`.
