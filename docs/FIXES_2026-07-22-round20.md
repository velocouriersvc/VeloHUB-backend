# Round 20 - 2026-07-22

Tester feedback after the Round 19 deploy. Three issues reported, all root-caused against the code
rather than guessed.

## Issue 2 - "Drivers code was unable to be verified: Action Failed"

**Root cause: a route contract mismatch, not the code comparison.**

`merchantRoutes.ts` guarded `/orders/:orderId/verify-pickup` with
`body("pickupCode").required().isString().minLength(6)`. The app sends `{ code }` and the controller
reads `code`, so the validator rejected **every** request with 400 before the controller ever ran.
Even with the right field name, `minLength(6)` would have rejected the 4-digit driver PIN. The
sibling `/complete-pickup` route has no validator at all, which is exactly why store pickup worked
and delivery never could.

The error was unreadable for a second, independent reason: the seller screen read
`error.response.data.message` (an axios shape) while `ApiClient` is fetch-based and puts the message
on `error.message`. Every failure collapsed into "Something went wrong", so a genuine wrong code and
a validation rejection looked identical.

Fixed:
- validator is now `body("code").required().isString().minLength(4)`, accepting the 4-digit driver PIN
  and the 6-char store code
- `validate()` returns a top-level `message`, so validation failures are legible to every client
- the seller screen reads `error.message`
- `verifyPickupCode` post-verify side effects (audit, socket, notifications, SMS) are now best effort.
  They ran unguarded **after** the order was saved as PICKED_UP, so a notification failure reported a
  false failure on a correct code and left the merchant unable to retry, because the status guard
  rejects an already-picked-up order. Also removes a redundant second save.
- regression test pins the field name, both code lengths, and the message body

Checked but not changed: two other validator fields (`deliveryRating`, `estimatedPrepMinutes`) are
declared but unread. Both are `.optional()` and neither is ever sent by the app, so they are dead
rather than harmful.

## Issue 1.1 - Chat and call

Chat was already fully implemented (model, routes, service, screen) and working. The real defects:

- The **provider call button dialled `tel:undefined`**: it read `booking.customer?.phone ||
  booking.phoneNumber`, but `User` has `phoneNumber` (not `phone`) and `ServiceBooking` has no phone
  column at all. The app type declared a `phoneNumber` field the API never returns.
- The **customer call button silently disappeared** whenever a provider had never filled in
  `businessPhone`, with no fallback to their account phone.
- **Neither chat nor call was gated on acceptance**, client or server.

Fixed:
- `canContact(status)` helper: accepted / scheduled / in_progress only
- `getMessages` and `sendMessage` return 403 before acceptance; completed bookings stay readable so
  history is not lost
- `getMyBookings` falls back to the merchant's account phone when the profile has none
- both payloads expose `canContact` + `canCall` and only include a phone number once contact is
  permitted; both screens gate on those flags
- the phantom `phoneNumber` field is gone and the nested customer relation now says `phoneNumber`

Also fixed a latent import-time failure found by the new tests: `StripeProvider` built its SDK client
in the constructor and the provider registry instantiates every provider at import, so importing
anything that reached `payment-service` threw "Neither apiKey nor config.authenticator provided"
wherever `STRIPE_SECRET_KEY` was unset. The client is now built on first use, matching Paystack.

## Issue 1.2 - Payment before the request reaches the merchant

`createBooking` notified the merchant immediately after payment **initiation**. For card and momo
that only means Paystack returned a checkout URL: the customer had not paid. `getMerchantBookings`
had no payment or status filter, so an unpaid booking appeared on the provider's list at once and
could be accepted.

Mirrors the existing order flow (`AWAITING_PAYMENT` plus `applyOrderPaidSideEffects`):
- new `ServiceBookingStatus.AWAITING_PAYMENT`; bookings are created in it
- `applyBookingPaidSideEffects` promotes to REQUESTED and sends the merchant their first
  notification, called from the payment side effects on confirmation. Idempotent, so a replayed
  webhook does not re-notify. Multi-date bookings promote every date but notify once.
- `getMerchantBookings` excludes AWAITING_PAYMENT; `updateStatus` refuses any transition while unpaid
- `reapUnpaidBookings` cancels abandoned checkouts after 15 minutes with **no refund** (nothing was
  captured) so they stop holding the provider's slot, wired into the existing expiry tick
- the app's second booking path used to ignore the Paystack URL and alert "Booking Confirmed" for a
  booking that was never paid; it now routes to the payment webview

## Issue 1.1 (calling) - BLOCKED

The requested Bolt/Yango model is masked calling: initiated in-app with neither party seeing the
other's number. Twilio secrets exist in the live cluster and are correctly shaped (SID `AC` + 34
chars, 32-char token, E.164 number), but they **do not authenticate**: the Twilio API returns
`401 / 20003 Authenticate`. The account is closed or the token was rotated. There is also no `twilio`
package and no Twilio code in the repo; Prelude (SMS/WhatsApp/verify only, no programmable voice)
replaced it.

Masked calling cannot be built or tested end to end until working voice credentials exist. Until
then chat is gated correctly and the call button dials the real number once contact is permitted.
