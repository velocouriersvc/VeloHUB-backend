## Round 9 - 2026-07-16: Paystack card checkout + currency + code gating

- Card form never appeared (blank "Secure payment" WebView): Paystack's hosted
  checkout needs DOM storage, cookies and window.open handling, which the
  WebView dropped silently. Added javaScriptEnabled/domStorageEnabled/
  sharedCookiesEnabled/thirdPartyCookiesEnabled, setSupportMultipleWindows
  false, a window.open same-window shim, onOpenWindow, and onError/onHttpError
  so failures show the retry screen instead of a black page.
- NGN cart charged as GHS: orders now resolve country/currency from the
  MERCHANT (products are priced in the merchant's currency), falling back to
  the customer then GH. Paystack charges are clamped to its supported
  currencies (NGN, GHS, ZAR, KES, USD, XOF, EGP); anything else falls back to
  USD. getQuote uses the same resolution so fees/tax follow the merchant market.
- Pickup code issued before payment: codes for ONLINE orders are now sent only
  when the payment is CONFIRMED (from the payment side effects); cash orders
  keep getting them at checkout. Unpaid online orders also have their codes
  BLANKED in every customer/merchant order read, and merchants cannot accept
  an online order until it is paid (auto-cancel reaps abandoned ones).
- "Ride Request" wording: delivery-type jobs now say "New Delivery Request"
  (driver push), "Delivery Requested" (customer), and "Delivery Request"
  (driver jobs list).
- Handshake copy simplified: driver pickup card "Give your code to the
  Merchant for verification"; merchant input "Enter code from the Driver
  before handoff"; customer card "Give your code to the Driver and collect
  your delivery"; driver drop-off modal "Enter code from the customer before
  handoff".

## Round 10 - 2026-07-16: currency-aware gateway, unpaid-order gating, no cash delivery

- Payment still failing on NGN carts ("Currency not supported by merchant"):
  the Ghana Paystack integration can only charge GHS/USD. Round 9 clamped
  against Paystack's full currency list, but the ACCOUNT is narrower. Now the
  supported set is env-driven (PAYSTACK_SUPPORTED_CURRENCIES, default
  "GHS,USD"); any charge in an unsupported currency is converted to USD using a
  new admin-editable usdExchangeRate (local units per 1 USD) on platform
  settings, seeded per country (GH 15.5, NG 1550, KE 129, ZA 18, etc.,
  PLATFORM_SETTINGS_SEED_VERSION bumped to 3). Only gateway charges (card/momo)
  convert; the order keeps its local prices, and metadata stashes
  originalAmount/originalCurrency/usdRate for reconciliation. Admin Fee
  Structure > General now edits the rate.
- Real provider errors surfaced: Paystack initialize/momo failures now return
  the provider message ("Currency not supported by merchant") instead of a
  generic "Payment initiation failed", so checkout tells the customer (and our
  logs) what actually went wrong.
- Unpaid order reached the merchant: online (momo/card) orders now start as a
  new AWAITING_PAYMENT status and only flip to PENDING once the payment
  confirms. The merchant list and dashboard exclude AWAITING_PAYMENT entirely,
  and the new-order notification/SMS + customer confirmation + pickup/delivery
  codes are ALL deferred to the payment side effects (applyOrderPaidSideEffects,
  idempotent). Abandoned unpaid orders are reaped after 10 min (stock restored,
  no penalty/refund since no money moved). Customers can still see and cancel
  their own awaiting-payment orders; the app timeline shows "Awaiting Payment"
  as the first step. Enum value added via idempotent boot sync
  (sync-order-status-enum.ts).
- No cash for deliveries: checkout rejects CASH when deliveryType is delivery
  ("Cash is not accepted for delivery orders"); the app hides the Cash option
  on delivery and auto-switches a cash selection to Mobile Money. Pickup orders
  and rides keep cash.
- Cart currency: the Basket screen hard-coded "$" on item price, subtotal and
  estimated total. Now uses the customer currency symbol (useCurrency), so a
  Ghana cart shows the cedi like the rest of the app.

Requires: backend deploy (enum sync + new nullable column are synchronize-safe;
set PAYSTACK_SUPPORTED_CURRENCIES if the live account supports more than
GHS,USD). Admin auto-deploys. ONE app change set (OTA-eligible, no new native
deps). Verification: backend tsc clean + 63/63 tests; app tsc clean; admin JSX
parses.

## Round 11 - 2026-07-17: Paystack in-app checkout finally renders

Video feedback confirmed round 10 works (correct GHS charge, merchant only
notified after payment) but the "Secure payment" WebView still rendered a
blank white page on iOS, so no one could enter card details. This was the
third round fighting the same WKWebView symptom: round 9 already added every
documented hardening prop (DOM storage, cookies, window.open shim,
onOpenWindow, error handlers) and Paystack's hosted checkout still refused to
render inside react-native-webview on the user's device.

Fix: stop using WKWebView for the checkout entirely. app/payment-webview.tsx
now opens the Paystack authorization URL in the system in-app browser via
expo-web-browser (SFSafariViewController on iOS, Chrome Custom Tabs on
Android), which is a real Safari/Chrome surface where Paystack officially
works (cookies, JS, popups, 3DS). The screen keeps its exact phase machine:

- While the sheet is open, the app polls GET /payments/status/:reference every
  3s (up to 5 min, card entry takes time). On success it AUTO-DISMISSES the
  sheet, shows "Payment successful" and routes to the order/ride/booking.
- If the customer closes the sheet first, a 60s "Confirming payment" window
  covers the webhook/callback race, then the existing timeout UI.
- Missing url/reference now fails visibly instead of sitting on a blank page.
- Failed/refunded polls dismiss the sheet and offer "Try again" (reopens).

One file covers all three verticals (orders, rides, service bookings) since
checkout.tsx, rides.tsx and services/product/[id].tsx all route here with the
same params. Backend untouched: callback page, status endpoint and payment
side effects were already proven. react-native-webview stays installed but is
no longer imported by any screen.

Requires: NO backend deploy, NO new EAS build (expo-web-browser has been in
the binary since before round 7). Ship as an EAS Update (OTA) or fold into the
next build. Verification: app tsc clean; em-dash sweep clean.

## Round 12 - 2026-07-17: GHS settlement charges, auto driver dispatch, service listing unblock

- NGN checkout STILL failed after round 10. Production logs proved the round-10
  conversion worked (NGN 1850 converted to USD 1.19 at rate 1550) and Paystack
  then rejected the USD charge itself with "Currency not supported by
  merchant": the live Ghana account has ONLY GHS enabled (USD needs Paystack
  approval). gatewayCharge now converts unsupported-currency charges into a
  SETTLEMENT currency instead of hardcoded USD: env PAYSTACK_SETTLEMENT_CURRENCY,
  defaulting to the first entry of PAYSTACK_SUPPORTED_CURRENCIES (new default
  "GHS"). Cross rate uses the admin-editable usdExchangeRate rows
  (local/USD divided by settlement/USD), so NGN 1850 charges as about GHS 18.50.
  Orders keep local prices; payment.metadata stores originalAmount,
  originalCurrency and fxRate. When Paystack enables USD or NGN on the account,
  just set PAYSTACK_SUPPORTED_CURRENCIES accordingly; no code change.
- "System should trigger driver request after merchant prepares the order":
  there was NO dispatch at all. merchant-service.updateOrderStatus only
  notified the customer on READY_FOR_DELIVERY, and no driver screen ever
  listed the ready pool (the backend endpoint existed but had zero app
  callers; the driver Jobs screen only fetched rides). Two-part fix:
  1) Backend: on READY_FOR_DELIVERY, dispatchDeliveryDrivers finds nearby
     online drivers (all vehicle tiers, 20 km, via the existing
     DriverMatchService) from the merchant's coordinates and broadcasts the
     existing "New Delivery Request" push + ride:new socket event with
     kind order_delivery. Best-effort: dispatch failure never blocks the
     status change. No ride row and no extra charge are created; the driver
     claims the ORDER through the existing acceptDelivery flow (4-digit PIN
     handshake, POD, settlement from the order's deliveryFee).
  2) Driver app: Jobs now also lists prepared delivery orders (merchant name
     as pickup, customer address as dropoff, deliveryFee as the earning) and
     accepting one calls acceptDelivery then opens the Deliveries flow. The
     existing ride:new socket listener and 10s poll refresh the list, so the
     broadcast surfaces immediately on both nearby phones and disappears for
     the second driver once the first accepts (Redis lock prevents
     double-claims).
- Service listing was impossible: "Please choose the type of service" fired on
  every submission because add-product.tsx validated a serviceType field whose
  selector was removed long ago (nothing ever set it). The dead check is gone;
  category (which now captures the service subtype, e.g. Beauty) is required
  for BOTH listing kinds, and service tags derive from the category. Note: the
  "listed in the hire section after approval" copy is cosmetic; services go
  live immediately.

Requires: backend deploy (no schema changes; PAYSTACK_SUPPORTED_CURRENCIES env
now defaults to GHS so no k8s change needed). App changes are OTA-eligible (no
new native deps). Verification: backend tsc clean + 63/63 tests; app tsc
clean; em-dash sweep clean; prod-log diagnosis of the NGN failure captured
before the fix.

## Round 13 - 2026-07-18: global currency preference + invisible service listings

Feedback from a US-based seller (Solomon): seller screens (dashboard Total
Sales / Available Balance, Inventory Tracker, New Service fee field) showed
GH cedi for a US account while other screens showed USD, and the service he
listed never appeared in Professional Services discovery.

Currency (app):
- useCurrency was rewritten around an explicit resolution order:
  1) manual "Default Currency" chosen in Settings (new, persisted),
  2) profile / signup country, 3) DEVICE LOCALE region (a US phone resolves
  USD with zero setup), 4) GH fallback. The old fire-and-forget GPS reverse
  geocode branch is gone; resolution is synchronous and consistent on every
  screen, which removes the "GHS here, USD there" mismatch.
- New "Default Currency" picker (Automatic + 11 currencies) in BOTH the
  customer Profile > Settings section and the seller Company Settings >
  Currency card. The preference lives in a persisted zustand store, so
  changing it re-renders every mounted price label instantly (no restart).
- Seller screens that bypassed the resolver (finances, orders, order-details,
  transaction-history passed the raw signup country into formatPrice) now use
  the active currency; order screens prefer the ORDER's own stored currency so
  historical orders never re-label amounts.
- Data integrity note: prices were already stored as numeric decimals backend
  side (no symbols in data); display symbols are injected at render time only.

Service discovery (backend, the "my listing does not show" bug):
- getProducts filtered merchant.country = :country, so merchants with NO
  country set were invisible in EVERY market; now NULL-country merchants are
  included everywhere until they set one.
- The 20 km radius filter required merchantProfile coordinates to exist,
  hiding every listing of a provider who never set a location. Radius checks
  are now fail-open for missing coordinates (both product listing and the
  general product search).
- With the device-locale fix the app also stops sending country=GH for
  non-Ghana users, so a US buyer sees US (and unset-country) providers.

Requires: backend deploy (query changes only, no schema). App changes are
OTA-eligible. Verification: backend tsc + 63/63 tests; app tsc clean; em-dash
sweep clean.

## Round 14 - 2026-07-18: services separation, per-listing call types, booking fees, driver navigation

Marketplace separation + service stock concepts:
- Service listings leaked into the Marketplace screen (screenshots showed the
  same "High Fade" in both surfaces): the marketplace fetched ALL products.
  The backend "product" category type now expands to every NON-service
  category and the Marketplace screen requests category=product, so services
  only live under Professional Services.
- The Inventory list marked services "LOW STOCK / 0 units left" and
  edit-product showed stock inputs for them, because "is a service" was
  derived from the MERCHANT's business type (or the literal 'services' slug).
  Both now detect services per ITEM via the category's type (with a
  serviceDurationMin fallback): inventory shows a SERVICE badge + duration and
  "Manage Availability"; editing a beauty/cleaning/etc. service hides stock.

Per-listing In-Call / Out-Call (end to end):
- Product gains inCall (default true) / outCall (default false). Add/Edit
  service shows two multi-select cards ("At My Location" / "In-Home / Mobile");
  at least one is required (client + server validation), and in-call listings
  require the merchant to have a business address + coordinates.
- The booking screen's call-type tabs are now driven by the LISTING's flags
  (profile switches remain the fallback for old rows), quoteBooking enforces
  the same, and the service detail shows plain-language badges.
- Address visibility both ways: in-call bookings show the PROVIDER's address
  to the customer (booking screen "Service location" card + My Bookings rows
  via a new providerAddress field on getMyBookings); the provider's bookings
  card now shows an In-Call / In-Home badge and only shows the customer's
  address + map link for in-home jobs (in-call says "Customer comes to your
  location" instead of an empty pin).

Service booking platform fee (customer-paid, itemized):
- platform_settings gains serviceBookingFeeRate (%); fee per booked date =
  serviceBookingFee (fixed) + price * rate / 100. Both editable in admin Fee
  Structure > Services > Booking & Cancellation Fees.
- quoteBooking returns platformFee; createBooking stores it on the booking
  (new platformFee column) and the combined payment total is
  (price + travelFee + platformFee) x dates. The app breakdown itemizes
  "Platform fee x N" for the customer.
- Settlement fix (found during this round): providers were NEVER paid their
  travel fee (commission also only ever applied to price). Provider payout is
  now price - commission + travelFee (travel fee passes through
  uncommissioned); the customer booking fee stays platform revenue and is
  recorded in the settlement breakdown.

Driver navigation + auto-surfaced requests:
- GET /driver/deliveries/active now flattens merchantName / pickupAddress /
  pickupLat / pickupLng from the merchant profile. This ALSO fixes the in-app
  navigation screen routing deliveries to 0,0 (it read those flat fields that
  never existed).
- The active delivery card shows the merchant name + pickup address and gains
  "Navigate to Merchant" (before the PIN handshake) and "Navigate to
  Customer" (after pickup) buttons that open the existing in-app turn-by-turn
  navigation screen.
- The driver dashboard now auto-surfaces incoming requests: socket push +
  15s poll render an "Incoming Requests (N)" floating card (distinct
  Delivery / Ride badges, fare, pickup; tap opens the jobs list). No more
  requests hiding behind the Find Opportunities button.

Requires: backend deploy (new defaulted columns are synchronize-safe; no enum
changes). Admin auto-deploys. App changes are OTA-eligible (no new native
deps). Verification: backend tsc + 63/63 tests; app tsc clean; admin JSX
