## Round 9 - 2026-07-16: Paystack card checkout + currency + code gating

- Card form never appeared (blank "Secure payment" WebView): Paystack's hosted
  checkout needs DOM storage, cookies and window.open handling, which the
  WebView dropped silently. Added javaScriptEnabled/domStorageEnabled/
  sharedCookiesEnabled/thirdPartyCookiesEnabled, setSupportMultipleWindows
  false, a window.open same-window shim, onOpenWindow, and onError/onHttpError
  so failures show the retry screen instead of a black page.
- NGN cart charged as GHS: orders now resolve country/currency from the
  MERCHANT (products are priced in the merchant's currency), falling back to
  the customer then GH. Paystack charges are clamped to its supported
  currencies (NGN, GHS, ZAR, KES, USD, XOF, EGP); anything else falls back to
  USD. getQuote uses the same resolution so fees/tax follow the merchant market.
- Pickup code issued before payment: codes for ONLINE orders are now sent only
  when the payment is CONFIRMED (from the payment side effects); cash orders
  keep getting them at checkout. Unpaid online orders also have their codes
  BLANKED in every customer/merchant order read, and merchants cannot accept
  an online order until it is paid (auto-cancel reaps abandoned ones).
- "Ride Request" wording: delivery-type jobs now say "New Delivery Request"
  (driver push), "Delivery Requested" (customer), and "Delivery Request"
  (driver jobs list).
- Handshake copy simplified: driver pickup card "Give your code to the
  Merchant for verification"; merchant input "Enter code from the Driver
  before handoff"; customer card "Give your code to the Driver and collect
  your delivery"; driver drop-off modal "Enter code from the customer before
  handoff".

## Round 10 - 2026-07-16: currency-aware gateway, unpaid-order gating, no cash delivery

- Payment still failing on NGN carts ("Currency not supported by merchant"):
  the Ghana Paystack integration can only charge GHS/USD. Round 9 clamped
  against Paystack's full currency list, but the ACCOUNT is narrower. Now the
  supported set is env-driven (PAYSTACK_SUPPORTED_CURRENCIES, default
  "GHS,USD"); any charge in an unsupported currency is converted to USD using a
  new admin-editable usdExchangeRate (local units per 1 USD) on platform
  settings, seeded per country (GH 15.5, NG 1550, KE 129, ZA 18, etc.,
  PLATFORM_SETTINGS_SEED_VERSION bumped to 3). Only gateway charges (card/momo)
  convert; the order keeps its local prices, and metadata stashes
  originalAmount/originalCurrency/usdRate for reconciliation. Admin Fee
  Structure > General now edits the rate.
- Real provider errors surfaced: Paystack initialize/momo failures now return
  the provider message ("Currency not supported by merchant") instead of a
  generic "Payment initiation failed", so checkout tells the customer (and our
  logs) what actually went wrong.
- Unpaid order reached the merchant: online (momo/card) orders now start as a
  new AWAITING_PAYMENT status and only flip to PENDING once the payment
  confirms. The merchant list and dashboard exclude AWAITING_PAYMENT entirely,
  and the new-order notification/SMS + customer confirmation + pickup/delivery
  codes are ALL deferred to the payment side effects (applyOrderPaidSideEffects,
  idempotent). Abandoned unpaid orders are reaped after 10 min (stock restored,
  no penalty/refund since no money moved). Customers can still see and cancel
  their own awaiting-payment orders; the app timeline shows "Awaiting Payment"
  as the first step. Enum value added via idempotent boot sync
  (sync-order-status-enum.ts).
- No cash for deliveries: checkout rejects CASH when deliveryType is delivery
  ("Cash is not accepted for delivery orders"); the app hides the Cash option
  on delivery and auto-switches a cash selection to Mobile Money. Pickup orders
  and rides keep cash.
- Cart currency: the Basket screen hard-coded "$" on item price, subtotal and
  estimated total. Now uses the customer currency symbol (useCurrency), so a
  Ghana cart shows the cedi like the rest of the app.

Requires: backend deploy (enum sync + new nullable column are synchronize-safe;
set PAYSTACK_SUPPORTED_CURRENCIES if the live account supports more than
GHS,USD). Admin auto-deploys. ONE app change set (OTA-eligible, no new native
deps). Verification: backend tsc clean + 63/63 tests; app tsc clean; admin JSX
parses.

## Round 11 - 2026-07-17: Paystack in-app checkout finally renders

Video feedback confirmed round 10 works (correct GHS charge, merchant only
notified after payment) but the "Secure payment" WebView still rendered a
blank white page on iOS, so no one could enter card details. This was the
third round fighting the same WKWebView symptom: round 9 already added every
documented hardening prop (DOM storage, cookies, window.open shim,
onOpenWindow, error handlers) and Paystack's hosted checkout still refused to
render inside react-native-webview on the user's device.

Fix: stop using WKWebView for the checkout entirely. app/payment-webview.tsx
now opens the Paystack authorization URL in the system in-app browser via
expo-web-browser (SFSafariViewController on iOS, Chrome Custom Tabs on
Android), which is a real Safari/Chrome surface where Paystack officially
works (cookies, JS, popups, 3DS). The screen keeps its exact phase machine:

- While the sheet is open, the app polls GET /payments/status/:reference every
  3s (up to 5 min, card entry takes time). On success it AUTO-DISMISSES the
  sheet, shows "Payment successful" and routes to the order/ride/booking.
- If the customer closes the sheet first, a 60s "Confirming payment" window
  covers the webhook/callback race, then the existing timeout UI.
- Missing url/reference now fails visibly instead of sitting on a blank page.
- Failed/refunded polls dismiss the sheet and offer "Try again" (reopens).

One file covers all three verticals (orders, rides, service bookings) since
checkout.tsx, rides.tsx and services/product/[id].tsx all route here with the
same params. Backend untouched: callback page, status endpoint and payment
side effects were already proven. react-native-webview stays installed but is
no longer imported by any screen.

Requires: NO backend deploy, NO new EAS build (expo-web-browser has been in
the binary since before round 7). Ship as an EAS Update (OTA) or fold into the
next build. Verification: app tsc clean; em-dash sweep clean.
