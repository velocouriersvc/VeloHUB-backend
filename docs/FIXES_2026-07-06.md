# Fixes - 2026-07-06

## 1. Ride requests not showing on the driver side  [FIXED]
Root cause: the driver app's `fetchJobs` polled `getActiveRide()` - the ride already
ASSIGNED to that driver. A freshly requested ride is `SEARCHING` with `driverId = null`,
so it was never returned; drivers saw nothing. Dispatch was socket/push-only (no way to
poll pending requests), and the match was strict (online-in-Redis + APPROVED + exact
vehicle type).

Fix:
- Backend: new `GET /driver/rides/available` (`ride-service.getAvailableRides`) returns
  recent (last 5 min), unassigned, still-searching rides with a loose 50km proximity
  filter and NO vehicle-type / approval / online gating. `acceptRide` already only
  requires the ride to be searching, so any driver who sees one can accept it.
  Files: `src/services/ride-service.ts`, `src/controllers/DriverController.ts`,
  `src/routes/driverRoutes.ts`. Pushed to develop (`c4402ce`).
- App: `DriverRepository.getAvailableRides()` + `jobs.tsx` now polls available rides
  (every 10s and on the `ride:new` socket event) and shows them as job requests.
Deploy: backend needs develop -> main; app ships in the next EAS build.

## 2. Slow / failing fare calculation when scheduling a ride  [FIXED, app]
Root cause: the schedule modal called `resolveRoute` (Google Directions), a network
round-trip that could be slow or hang, so the fare sometimes never appeared.

Fix (`VeloHUB/components/rides/ScheduleRideModal.tsx`, `VeloHUB/utils/route.ts`):
- Schedule fare now uses the instant haversine `estimateRoute` (no Directions call) -
  a scheduled fare is an estimate and is re-confirmed when the ride starts.
- Added a 5s AbortController timeout to the Places detail fetch and to `resolveRoute`'s
  Directions fetch, so neither can stall the fare (also helps the live ride + checkout).
Ships in the next EAS build.
