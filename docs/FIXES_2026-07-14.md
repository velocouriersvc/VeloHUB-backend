# Round 7 - 2026-07-14

In-app payments, support thread, identity at signup, gateway-payments admin, a
native live-tracking widget, and the delivery-readiness epic (G1-G8).

## Slice 1 - In-app payments end to end (P0 money path)

Paystack payments succeeded at the gateway but the customer's browser landed on an
apiKey-403 JSON page because `callback_url` was never sent, and the webhook only
flipped the Payment row (never the ride/order). Fixed:

- `payment-service`: sets `callback_url` per transaction to a new public
  `GET /api/v1/payments/callback`; new `applyPaymentSideEffects` advances the Payment
  AND its ride/order idempotently (from both the callback and the webhook);
  `getPaymentByReference` replaces the load-all-pending scan.
- `GET /payments/callback` renders a friendly HTML result page (no more raw JSON).
- New `GET /payments/status/:reference` for in-app polling.
- App: new `app/payment-webview.tsx` (react-native-webview) loads the gateway page,
  intercepts the callback, polls status, and shows success/failed/dropped in-app;
  wired into `rides.tsx` and `checkout.tsx`; dead `velohub://` scheme fixed to `velo://`.

**Paystack dashboard config (user):** callback
`https://api.velocouriersvc.com/api/v1/payments/callback`, webhook
`https://api.velocouriersvc.com/api/v1/payments/webhook`.

## Delivery readiness (G1-G8)

- **G1** Driver-merchant handshake: `orders.driverPickupCode` (4-digit) minted on driver
  assignment; merchant verifies it (or scans the driver's QR) before the order can move
  to in-transit. Driver app shows the code + QR; merchant order-details has a QR scanner.
- **G2** Drop-off: on pickup verification the customer gets a delivery code by push + SMS
  (Prelude); the driver enters it (now 6-char) and a proof-of-delivery photo
  (`orders.podPhotoUrl`) is mandatory before completion.
- **G3** Store pickup: customer QR of the pickup code on the tracking screen; the
  merchant scanner releases/completes it.
- **G4** Merchant alerting + auto-cancel: seller dashboard loops an alert while orders
  are pending; `order-auto-cancel` job cancels PENDING orders older than 10 min, refunds
  to wallet, restores stock, and bumps `merchant_stats.autoCancelledOrders`.
- **G5** Driver off-boarding: `driver_profile.flaggedAt/flagReason`; admin flag/clear/list
  endpoints + flagged panel; `driver-flag-suspender` job suspends drivers flagged over 6h.
- **G6** Product variants: `product_variants` table (color/size/stock/priceDelta/image);
  merchant CRUD; per-variant stock; PDP selector with out-of-stock + low-stock rules;
  cart carries the variant; order snapshot + merchant receipt show the variant label.
- **G7** Product reviews: `product_reviews` table (verified-buyer, one per product/order);
  filtered list + summary on the PDP; POST guarded to delivered orders.
- **G8** Polish: `platform_settings.taxRate` + tax line at checkout; `cart_items.instructions`;
  prescription products blocked from the cart with a badge; promo/discount line at checkout.

## DB safety

All new tables (`support_ticket_messages`, `product_variants`, `product_reviews`) or new
nullable/defaulted columns (`orders.driverPickupCode/driverPickupVerifiedAt/podPhotoUrl`,
`driver_profile.flaggedAt/flagReason`, `merchant_stats.autoCancelledOrders`,
`cart_items.variantId/variantLabel/instructions`, `platform_settings.taxRate`) - safe for
synchronize:true, no varchar resizing or enum changes.
