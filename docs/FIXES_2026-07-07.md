# Fixes - 2026-07-07

## 1. Driver car icon does not move + "Driver Has Arrived"  [FIXED, app]
Root cause: on the driver's trip screen (`navigation.tsx`) `watchPositionAsync` only
updated local state - it never emitted the location to the socket. So during the trip
the customer received no `driver:moved` events and the car marker never moved.
- `app/(driver-tabs)/navigation.tsx`: streams the driver location via
  `useDriverSocket().emitLocation(lat, lng, heading, rideId)` on the initial fix and on
  every position update. The backend already broadcasts `driver:moved` to customers, and
  the customer's `RideTracking` already renders a rotating car marker at that location, so
  it now moves as the driver approaches.
- `app/rides.tsx`: shows a one-time "Driver Has Arrived" alert when the ride status
  becomes `arrived` (in addition to the existing arrival sound).

## 2. Call the driver  [FIXED, app]
The phone button on the customer's tracking screen had no handler.
- `components/rides/RideTracking.tsx`: the call button now dials the driver
  (`tel:${activeRide.driver.phoneNumber}`) after a confirm, and falls back to in-app chat
  when no number is available. The ride response already loads the driver relation.

## 3. Wrong vehicle/plate -> cancel + 100% refund  [FIXED]
`cancelRide` had a TODO and never refunded.
- Backend: `payment-service.refundRidePayment(rideId)` refunds a successful prepayment to
  the customer's wallet (marks it REFUNDED, auto-creates the wallet if missing, no-op for
  cash). `ride-service.cancelRide` takes a `fullRefund` flag; `RideController` reads it
  from the body.
- App: `RideTracking` shows "Wrong vehicle or plate? Cancel for a full refund" which
  cancels with `fullRefund: true` and the reason "Wrong vehicle / plate number".
Files: `payment-service.ts`, `ride-service.ts`, `RideController.ts`, `rides.tsx`,
`RideTracking.tsx`, `rides.res-req.ts`.

## 4. Stop generic images for new product listings  [FIXED]
The only source of generic images was the boot backfill in `run-seeds.ts`.
- It is now OPT-IN: it only runs when `ENABLE_PRODUCT_IMAGE_BACKFILL=true` (or via
  `npm run seed:product-images`). New listings keep the merchant's images (or none) and are
  never switched to a generic image.
- Confirmed with the Admin Dashboard: `velo-admin` ProductForm uploads real image URLs and
  the backend `createProduct` does not inject any image, so no generic image is added there.

## 5. Ride scheduling fare still not displaying  [FIXED, app]
Root cause: `pickupCoords` was initialised once from the `bias` prop, but the modal mounts
before the current location loads, so `bias` was `undefined`, `pickupCoords` stayed `null`,
and the fare effect (which needs both coords) never ran.
- `components/rides/ScheduleRideModal.tsx`: sync `pickupCoords` from `bias` once it arrives,
  and fall back to fetching the device location when the modal opens. Combined with the
  earlier instant-haversine estimate, the fare now computes as soon as a destination is
  picked.
