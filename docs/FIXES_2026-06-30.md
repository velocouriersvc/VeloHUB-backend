# Fixes - 2026-06-30 (user-reported batch)

## 1. Card payment for a ride returns an error
The ride card path now goes through Paystack (`setPaymentMethod` -> `processCardViaPaystack`
-> `transaction/initialize` -> `authorizationUrl`). The code path is correct and Paystack
is the registered provider for GH/NG with `initiateCardPayment`, so a card failure is a
**configuration** issue: `PAYSTACK_SECRET_KEY` must be a valid **live** key in production
(the example file ships `sk_test_xxx`) and the Paystack account must support the country
currency (GHS/NGN).
- App (`VeloHUB/app/rides.tsx`): the post-accept payment step now surfaces the exact
  backend message via `Alert` instead of failing silently, and tells momo users to approve
  the prompt on their phone. So any remaining Paystack error is now actionable, not a
  mystery. Card/momo open the Paystack `authorizationUrl`.

## 2. Increase all ride/delivery pricing by 20%
- Rides: `seed-vehicle-pricing.ts` applies a single `FARE_INCREASE = 1.2` multiplier over
  the static rates for every country/vehicle (base, per-km, per-min, minimum fare).
- Delivery: `seed-platform-settings.ts` scales `deliveryBaseFee` and `deliveryPerKmFee` by
  `1.2` for every country.
Both are applied to the static config on each upsert, so they are **idempotent** (re-running
the seeds never compounds the increase). Both seeds UPDATE existing rows, so the increase
takes effect on the next production deploy (boot seeds run automatically).

## 3. Delivery address autocomplete (like Ride)
The marketplace checkout (`VeloHUB/app/checkout.tsx`) already searched places while typing,
but the surrounding `ScrollView` was missing `keyboardShouldPersistTaps`, so tapping a
suggestion while the keyboard was open dismissed it instead of selecting it - it felt broken.
- Added `keyboardShouldPersistTaps="handled"` to the checkout ScrollView. Suggestions now
  appear while typing (vicinity-biased) and are selectable, exactly like the Ride flow.

## 4. Scheduling a ride crashed on Android + schedules not persisted
Root cause of the crash: `ScheduleRideModal` shared the live ride flow's state
(`selectedDestination`, `suggestions`, `setSelectedDestination`, ...). Selecting a
destination mutated that shared state, which drove the map/booking screen (and its
reanimated animations) behind the open native Modal - a reliable Android crash.
- `VeloHUB/components/rides/ScheduleRideModal.tsx`: made fully self-contained. It now uses
  its **own** `usePlaceSearch` instance and **own** local pickup/dropoff state, so opening or
  using it can never drive the screen behind it. The call site in `rides.tsx` was simplified
  to pass only `initialPickup`, `country`, `bias`, `currencySymbol`.
- Persistence: scheduled rides were local React state seeded with 2 hard-coded defaults, so
  user-created schedules vanished on remount (only the defaults showed under VIEW ALL). They
  are now stored in `AsyncStorage` (`velo.scheduledRides.v1`) and the hard-coded defaults
  were removed, so schedules survive app restarts.

## 5. Login: only OTP to sign back in + cannot fix a wrong number
- Wrong number trap: tapping back on the OTP screen returned to login, whose redirect effect
  immediately bounced the user back to OTP (phone set + status not verified) - the loop that
  forced a reinstall. The OTP back button (and a new "Wrong number? Change it" link) now call
  `clearAuth()` then route to `/login`, so the user can edit the number.
  (`VeloHUB/app/(auth)/otp-verification.tsx`)
- Sign-in methods: `login-password.tsx` is a real, working email+password login but was only
  reachable from onboarding/register/forgot-password, never the main login screen. Added a
  "Prefer email? Sign in with password" link to `login.tsx`. Apple sign-in remains.
