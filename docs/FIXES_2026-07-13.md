# Fixes - 2026-07-13 (Round 6)

Five production issues, root-caused and fixed end-to-end across backend, app, and admin.

## 1. Cash ride credited the driver instead of debiting commission

**Symptom**: Driver Maxine chose cash on a 12 GHS trip; her wallet showed +9 GHS. It
should be NEGATIVE (she holds the fare and owes the platform its commission).

**Root cause (two bugs)**:
1. The ride reached settlement with `paymentMethod = null` (rides created through
   checkout never had a method persisted), and settlement classified cash with
   `paymentMethod === 'cash'` only, so a null method fell into the ONLINE branch and
   credited the driver payout (12 x 0.75 = 9) for money the platform never collected.
2. Even correctly-classified cash rides could never produce a negative balance:
   `walletService.debit` threw on insufficient funds and `settleCashRide` swallowed the
   error, so commission was silently uncollected.

**Fix**:
- `settlement-service.ts`: a ride settles as ONLINE only when
  `paymentStatus === PAID && paymentMethod set && method !== cash`. Everything else
  (cash, null method, unpaid) settles as CASH: debit the driver's wallet for
  commission + rider service fee.
- `wallet-service.ts debit()`: new `allowNegative` flag; cash settlement uses it so the
  balance goes below zero (driver owes platform, reconciled at next top-up/cash-out).
  Missing wallets are auto-created and retried; failures now log at error level.
- `ride-service.ts` / `checkout-service.ts`: `requestRide` accepts an optional
  `paymentMethod` persisted at creation; checkout passes the order's method so
  checkout-created rides are classified correctly from birth.
- Data correction (applied in production, single DB transaction): the credited ride
  RIDE-C13420CF had a Paystack payment stuck at status=pending (never completed), so
  the +9.72 credit was money the platform never collected. Also found ride
  RIDE-1A225BFF: a completed cash ride whose 2.28 commission was silently swallowed
  by the old debit-throw. Correction: reverse 9.72, debit 2.28 + 2.28. Final balance
  -4.56 GHS with a four-row audit trail; ride C13420CF's method set to cash.

## 1b. Seed scripts were failing on EVERY boot (round-5 fares never seeded)

Pod logs showed `Seed scripts failed: value too long for type character varying(3)`.
The platform_settings global fallback row uses country "DEFAULT" (7 chars) in a
varchar(3) column; the insert failed each boot and runSeeds aborted, so the round-5
price book v2 and NG zones never reached production. Fixed: column widened to
varchar(8), and run-seeds.ts now isolates each seed so one failure cannot starve the
rest. Fare book verification runs after this deploys.

## 2. Raw "priority" shown instead of "Velo Priority"

- `fare-service.ts VEHICLE_DISPLAY_NAMES`: added `priority: "Velo Priority"` to
  US / CA / GH / NG (previously fell through to the raw enum string).
- App: new `backendVehicleLabel()` helper in `components/rides/constants.ts` maps
  backend types (bike/car/priority/suv/truck) to branded names; used in
  `app/orders/track.tsx` and `app/orders/history.tsx` (both rendered raw
  `vehicleType` before: "priority ride").
- Admin `FeeStructure.jsx`: label "Priority" renamed to "Velo Priority".

## 3. Driver pickup-code entry: inline + scrollable

- `app/(driver-tabs)/navigation.tsx`: the bottom sheet content is now wrapped in a
  ScrollView (max 55% of screen height, keyboardShouldPersistTaps) so every section is
  reachable - previously a fixed View, which is why the driver could not scroll.
- The passive "rider set a pickup code" hint is replaced with an INLINE code-entry card
  (6-char input + inline error). The Start Trip CTA submits the inline value; the popup
  Modal was removed entirely.

## 4. Terms of Service link was a 404

Every legal URL in the app pointed at pages that do not exist on the website
(`/privacy-policy`, `/terms-of-service`). The live site serves only
`https://velocouriersvc.com/privacy/`, with Terms at the `#terms` anchor.

- Customer profile "Terms of Service" now opens `https://velocouriersvc.com/privacy/#terms`.
- Driver profile: privacy link fixed to `/privacy/` + a new Terms of Service card.
- Seller company settings: privacy link fixed to `/privacy/`.

## 5. Support messages now reach the admin dashboard AND come back

The pipeline app -> `POST /support` -> `support_tickets` -> admin API already existed,
but the SupportTickets page was NOT linked in the admin sidebar (unreachable), the user
was never told when support replied, and the app had no way to see ticket status.

- Admin `Layout.jsx`: "Support Tickets" added to the sidebar navigation.
- Backend `admin-service.updateSupportTicket`: on status change/resolution the user now
  gets a push + in-app notification with the ticket number and resolution text.
- Backend: new `GET /support` returns the authenticated user's own tickets.
- App `app/support/index.tsx`: new "My tickets" section under the contact form showing
  ticket number, subject, status chip, and support's reply.
