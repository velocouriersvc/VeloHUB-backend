# Round 22 - 2026-07-23

Two payment defects reported after Round 21. Both root-caused against the code.

## Issue 1 - Scheduled rides must not offer cash

A scheduled ride shows a 3-hour free-cancellation window with a 70% late-cancel / no-show fee. That
fee can only be withheld from money collected up front, so cash makes the window unenforceable. The
scheduled flow is entirely separate from the immediate-ride flow (own screen, own `/rides/schedule`
endpoint, own service), so the shared `PaymentSelection` `hideCash` prop did not apply.

- App `ScheduleRideModal.tsx`: cash removed from the local payment method list; the button always
  reads "Pay X and Schedule".
- Backend `ScheduledRideService.create`: rejects a missing or cash method ("Scheduled rides must be
  prepaid by card or mobile money"), no longer defaults to cash, and the cash short-circuit is gone,
  so every scheduled ride is charged up front. `ScheduledRideController.create` drops the `|| "cash"`
  default.

### The 70% policy was displayed but never enforced (now fixed)

`ScheduledRideService.cancel` always fully refunded, with no time check, so the policy shown to users
was fiction. It now mirrors the service-booking rule:

- Cancelling more than 3 hours before pickup: full refund.
- Within 3 hours (or a no-show): keep 70% of the fare, refund 30% to the wallet.
- `refundScheduledRidePayment` takes an optional partial amount; `cancel` returns
  `{ refunded, feeKept, late }`; the app's cancel dialog warns about the fee and the success alert
  shows the amount kept vs refunded.

## Issue 2 - Package payment "keeps buffering / never loads"

Definitive root cause: the package (Velo Send) checkout never forwarded the sender's phone number, so
MOBILE MONEY packages threw and never opened Paystack (Ghana is momo-heavy, so this was the dominant
symptom). Card packages worked because the email defaults.

- App builds the package payload with no sender phone -> `PackageRideCheckoutInput` has no phone
  field -> `checkout-service` calls `requestRide` without `phoneNumber`/`email` -> `processMomoPayment`
  throws "Phone number is required for momo payment" -> `requestRide` deletes the ride and rethrows ->
  no `authorizationUrl` is ever returned.

Fix:
- `PackageRideCheckoutInput` gains `phoneNumber` + `email`; the `package_ride` branch forwards them to
  `requestRide` (mirroring how the passenger prepaid flow forwards the body phone/email).
- The app package payload sends the authenticated sender's `phoneNumber` (`userPhone`) and `email`.
- The checkout response now lifts `authorizationUrl` + `paymentReference` to the top level (kept on
  `ride` too) so the payment webview always has a matching reference, preventing the secondary
  "polls a 404 for 5 minutes" buffering.
