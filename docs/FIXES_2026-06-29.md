# Fixes - 2026-06-29 (user-reported issues)

## Images not displaying - PRIMARY ROOT CAUSE (app)
Add/Edit Product saved the image picker's local `file://` URIs straight into the
product (`images`) WITHOUT uploading them. Those paths only exist on the seller's
phone, so images showed for the seller (and briefly on iOS while the cache held)
but were blank for every other user and after the cache cleared.
Fix (app, ships in next EAS build):
- `VeloHUB/app/(seller-tabs)/add-product.tsx` and `edit-product.tsx` now upload each
  local image via `uploadFile(..., category: "products")` and save the returned
  public URLs. Already-remote (http/https) URLs are passed through unchanged.
- `VeloHUB/repositories/upload-repository.ts`: added `products`/`merchants` categories.
Existing products saved with file:// paths stay blank until the seller edits and
re-saves them (which now uploads the images).

## Images not displaying - SECONDARY cause (serving)
The public image URL is `https://api.velocouriersvc.com/<bucket>/<key>`, but nginx
only proxies `location /` to the API (port 30080). It does NOT route the bucket
path to MinIO, and MinIO is internal-only (`minio-service:9000`). So image URLs
landed on the API, which had no route to serve them -> 404 -> blank everywhere.

Fix (API serves the assets itself; no nginx change, no URL migration):
- New public route streams the object from internal MinIO at the bucket path:
  `src/routes/publicAssetRoutes.ts`, mounted at `/<bucket>` in `src/index.ts`
  (before auth/api-key middleware). Existing image URLs now resolve.
- `ensureBucket` now ALWAYS (re)applies the public-read policy, not only on first
  creation, so an existing bucket is not left private (`src/utils/minio-client.ts`).
- Product image URLs are normalized to the public host on read (`toPublicImages`
  in `product-service.ts`) - from the prior change.
- App: `android.usesCleartextTraffic=true` (`VeloHUB/app.json`) - from the prior change.

## Driver Profile setup "internal server error" (ROOT CAUSE + clarification)
In the original code the transaction COMMITS (user, identification, driver profile,
role are all saved) and THEN sends a notification. The `notification_type_enum` in
the DB was missing values like `profile_created`, so the notify INSERT threw AFTER
the commit, the outer catch ran `rollbackTransaction()` on an already-committed
transaction, and the request returned 500 - even though the driver profile WAS
created. So users saw 500 on every attempt.

Fix (already on develop, needs deploy to take effect):
- Profile-setup notifications are best-effort (a notify failure can never 500 a
  committed setup) - `src/services/profile-service.ts`.
- `notification_type_enum` is synced on boot - `src/scripts/sync-notification-enum.ts`.
- Driver fields are truncated to their column lengths so a long value (e.g. a full
  address in `region` varchar(100)) cannot overflow.
These require merging develop -> main (production still runs the old code).
